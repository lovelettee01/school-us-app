# AGENTS.md

## 목적
- 이 저장소는 **React + TypeScript + Next.js + TailwindCSS + Zustand** 기반 애플리케이션을 AI 코딩 에이전트로 개발/유지보수한다.
- 모든 변경은 "빠른 수정"보다 **정확성, 유지보수성, 예측 가능성**을 우선한다.

## 기술 스택
- 프레임워크: Next.js (별도 지시가 없으면 App Router 기본)
- 언어: TypeScript (`strict` 모드)
- UI: React + TailwindCSS
- 상태 관리: Zustand
- 패키지 매니저: 기존 lockfile 우선 (`pnpm-lock.yaml` > `package-lock.json` > `yarn.lock`)

## 핵심 작업 원칙
1. 먼저 이해하고, 그다음 수정한다
- 수정 전에 관련 파일/타입/호출 경로를 읽고 맥락을 파악한다.
- 요청 범위에 집중해 최소 변경으로 해결한다.
- 명시적으로 요청되지 않은 리팩터링은 하지 않는다.

2. 변경 요청이 없으면 기존 동작을 유지한다
- API 계약, 라우트 구조, 컴포넌트 props 호환성을 깨지 않는다.
- 동작 변경이 필요하면 변경 이유와 영향 범위를 명확히 설명한다.

3. 항상 검증한다
- 수정 후 목적에 맞는 검증을 수행한다.
- 최소 검증 기준: 타입 체크 + 린트.
- 동작/데이터 흐름 변경이 있으면 관련 테스트를 실행한다.
- 런타임 영향이 있는 변경은 빌드 또는 해당 경로 수동 검증을 수행한다.
- 검증을 실행하지 못했다면 사유와 대체 확인 방법을 남긴다.

4. 명확하게 공유한다
- 무엇을, 왜 바꿨고, 어떻게 검증했는지 요약한다.
- 가정, 리스크, 후속 작업을 함께 명시한다.

5. 모든 코드에 주석은 상세하게 한국어로 작성한다
- 구현 코드에 목적, 설명, 파라미터, 필수여부 등 내용을 작성한다.

## 프로젝트 구조 기본 규칙
- `src/app`: Next.js 라우트, 레이아웃, 기본 서버 컴포넌트
- `src/components`: 재사용 UI 컴포넌트
- `src/features`: 기능 단위 컴포넌트/훅/스토어/셀렉터
- `src/lib`: 공용 유틸, API 클라이언트, 헬퍼
- `src/store`: 전역 Zustand 스토어(기능 분리가 어려운 경우)
- `public`: 정적 리소스

실제 저장소 구조가 다르면 이 기본값보다 기존 규칙을 우선한다.

## 네이밍 규칙
- 컴포넌트/타입: `PascalCase`
- 함수/변수/훅: `camelCase` (`useXxx` 패턴)
- 디렉터리/파일: `kebab-case`
- 불리언 변수: `is/has/can` 접두어 사용
- 이벤트 핸들러: `handleXxx`, 파생값: `derivedXxx` 권장

## Next.js + React 가이드
- 정적이거나 읽기 중심 UI는 Server Component를 우선 사용한다.
- 다음 경우에만 Client Component(`use client`)를 사용한다.
    - 브라우저 전용 API 사용
    - 로컬 인터랙션 상태/이벤트 처리
    - UI에서 Zustand 스토어 사용
- 라우트 단위 `loading.tsx`, `error.tsx`는 사용자 체감이 큰 경로부터 우선 적용한다.
- 데이터 패칭은 라우트 경계에 가깝게 두고 불필요한 클라이언트 패칭을 피한다.
- Next.js 캐시/재검증 기본 동작을 존중하며, 기본값이 아닌 선택에는 근거를 남긴다.
- 서버 액션/핸들러에서 예외는 사용자 메시지와 로깅 관점으로 분리해 처리한다.

## TypeScript 가이드
- `strict` 호환을 유지한다.
- `any` 사용을 피하고 명시적 타입/제네릭/구별된 유니온을 우선한다.
- 외부 입력(API, query params, env)은 `unknown`으로 받은 뒤 narrowing/검증한다.
- 타입 단언(`as`)은 최소화하고, 필요한 경우 안전 근거를 코드 근처에 남긴다.
- 공유 도메인 타입은 기능 단위 단일 진입점에서 export한다.

## Zustand 가이드
- 전역 상태가 아니면 기능 범위 store(per feature)로 관리한다.
- 스토어 상태는 작고 직렬화 가능하게 유지한다.
- 컴포넌트에서 전체 스토어 선택을 피하고 selector를 기본 사용한다.
- 비동기 부수효과는 스토어 액션 또는 서비스 레이어 중 하나로 일관되게 둔다.
- 직접 mutation 대신 불변 업데이트 패턴을 사용한다.
- 스토어 공개 API는 `state + actions` 형태를 유지한다.

## TailwindCSS 가이드
- 유틸리티 클래스는 컴포넌트에 직접 적용하고, 반복이 의미 있을 때만 추출한다.
- 클래스 순서는 읽기 쉽게 유지한다(레이아웃 > 간격 > 타이포 > 색상 > 효과).
- 일회성 arbitrary value 남용보다 디자인 토큰/테마 확장을 우선한다.
- 반응형(`sm/md/lg/xl`)과 접근 가능한 색 대비를 보장한다.
- Tailwind로 표현 가능한 경우 인라인 스타일을 피한다.

## UI/UX 및 접근성
- 키보드 접근성과 명확한 포커스 상태를 보장한다.
- 시맨틱 HTML을 우선하고 ARIA는 필요한 경우에만 정확히 사용한다.
- 비동기 UI에는 로딩/빈 상태/에러 상태를 제공한다.
- 클릭 타겟, 라벨, 에러 메시지는 스크린리더 기준으로 검토한다.

## API 및 데이터 안전
- 시크릿/토큰을 하드코딩하지 않는다.
- 환경 변수는 검증된 설정 헬퍼를 통해 사용한다.
- 네트워크/API 실패에 대해 사용자 친화적인 에러 처리를 구현한다.
- 서버 전용 값이 클라이언트 번들에 노출되지 않도록 한다.

## 테스트 및 품질
- 변경 지점과 가까운 테스트를 우선한다.
    - 유틸/스토어 로직: 단위 테스트
    - 상호작용 핵심 UI: 컴포넌트 테스트
    - 라우트/데이터 경계: 통합 테스트
- 버그 수정 시 가능하면 회귀 테스트를 추가한다.
- 스냅샷은 의도를 확인한 경우에만 업데이트한다.
- 테스트가 없으면 최소 재현 절차(입력, 기대 결과)를 작업 보고에 남긴다.

## 성능 가이드
- 불필요한 리렌더를 방지한다(필요 근거가 있을 때 memoization/selector 사용).
- 무거운 클라이언트 전용 모듈은 동적 import를 고려한다.
- Next.js 내장 기능으로 이미지/폰트 최적화를 우선한다.
- 사전 최적화보다 측정 기반 개선을 우선한다.

## 금지 사항
- 명시 요청 없이 파괴적 명령(광범위 삭제, 강제 리셋 등)을 실행하지 않는다.
- 요청 없이 대규모 의존성 업그레이드를 진행하지 않는다.
- 사전 합의 없이 호환성 깨는 변경을 도입하지 않는다.
- 테스트/명령 실행 결과를 허위로 작성하지 않는다.

## 테스트 플로우 (표준)
아래 순서를 기본으로 사용한다. 범위 밖 단계는 생략 가능하되 생략 사유를 보고에 남긴다.

1. 사전 확인
- 사용 패키지 매니저 확인(lockfile 기준)
- 변경 범위 식별(파일, 기능, 라우트)
- 영향도 분류
    - 타입 영향만 있음
    - UI/상태 로직 영향
    - 빌드/런타임 영향

2. 정적 검증 (필수)
- `typecheck` 실행
- `lint` 실행
- 실패 시 코드 수정 후 동일 단계 재실행

3. 테스트 실행 (조건부 필수)
- 해당 기능의 단위/컴포넌트/통합 테스트 우선 실행
- 관련 테스트가 없으면 최소 범위 테스트를 추가하거나, 미추가 사유를 기록

4. 빌드/런타임 확인 (영향 있을 때)
- 라우트/데이터 패칭/설정 변경 시 `build` 확인
- 필요 시 수동 시나리오 검증
    - 진입 경로
    - 사용자 액션
    - 기대 결과
    - 에러/로딩/빈 상태

5. 실패 대응 규칙
- 에러는 원인 분류 후 가장 작은 수정으로 해결
- 연쇄 실패가 나면 마지막 성공 지점부터 재검증
- 외부 요인(CI, 네트워크, 권한)으로 실패 시 재현 조건과 로그 요약 기록

6. 완료 보고 템플릿
- 변경 요약
- 수정 파일 목록
- 실행한 검증 단계/명령/결과
- 미실행 검증과 사유
- 리스크/가정
- 필요 시 다음 단계

## 작업 완료 보고 형식 (에이전트 응답)
작업 완료 시 아래 항목을 제공한다.
1. 변경 요약
2. 수정 파일 목록
3. 실행한 검증 단계와 결과
4. 리스크 / 가정
5. 다음 단계 제안(필요 시)

## Git 및 변경 관리 규칙 (선택사항)
- Git처리는 요청시만 진행한다.
- 커밋은 작고 원자적으로 유지한다.
- 명시적 요청 없이 히스토리를 재작성하지 않는다.
- 관련 없는 로컬 변경사항을 임의로 되돌리지 않는다.
- 예상치 못한 파일 변경을 발견하면 즉시 보고하고 진행 여부를 확인한다.

## 빠른 명령 체크리스트
저장소에서 이미 사용하는 패키지 매니저를 따른다.

- 의존성 설치: `pnpm install` / `npm install` / `yarn install`
- 개발 서버: `pnpm dev` / `npm run dev` / `yarn dev`
- 타입 체크: `pnpm typecheck` / `npm run typecheck` / `yarn typecheck`
- 린트: `pnpm lint` / `npm run lint` / `yarn lint`
- 테스트: `pnpm test` / `npm run test` / `yarn test`
- 빌드: `pnpm build` / `npm run build` / `yarn build`

필요한 스크립트가 없으면 요청된 작업 범위에서 필요한 최소 항목만 추가한다.
