# UI 리팩토링 실행 계획서

## 문서 목적
- 본 문서는 **구현 이전 계획 확정**을 위한 문서다.
- 이번 리팩토링의 목표는 다음 2가지다.
  - 공통 컴포넌트 체계를 구축해 화면 전반에서 일관된 UI를 사용한다.
  - 에러 메시지 표현을 인라인(컴포넌트 하단) 중심에서 전역 메시지(토스트/팝업) 중심으로 전환한다.

## 진행 상태 규칙
- `- [x]` : 미착수
- `- [x]` : 완료
- 각 Phase는 **체크리스트 항목이 모두 완료되면 실행 완료**로 판단한다.

## 전체 범위
- 포함
  - 버튼, 타이포그래피, 모달, 메시지, 배지 포함 공통 컴포넌트 표준화
  - 컴포넌트 분류 체계/Props 규칙/접근성 기준 수립
  - 에러 메시지 전역 시스템 설계(토스트/메시지 팝업 선택 가능)
  - 기존 에러 지점의 마이그레이션 전략 수립
- 제외
  - 실제 컴포넌트 구현 코드 작성
  - API 계약 변경
  - 대규모 라우트/비즈니스 로직 리팩토링

### 체크리스트
- [x] 컴포넌트 분류 체계 확정

- [x] 상태 모델 정리 (`default/loading/disabled/error/success` 등)

## Phase 3. 메시지 시스템 정책 설계 (핵심)
### 목표
- 에러 메시지를 전역 메시지 체계로 통일하고, 토스트/팝업 선택 가능 구조를 정의한다.

### 체크리스트
- [x] 메시지 타입 정의 (`error/warning/success/info`)
- [x] 메시지 모드 정의 (`toast/popup/auto`)
- [x] 기본 모드 결정 방식 설계 (전역 설정 + 이벤트별 override)
- [x] 중복 메시지 억제 정책 정의 (dedupe key / time window)
- [x] 우선순위 정책 정의 (`error > warning > success > info`)
- [x] 자동 닫힘 및 표시 시간 정책 정의
- [x] 사용자 액션(재시도/확인) 버튼 정책 정의
- [x] 접근성 정책 정의 (`aria-live`, 포커스 트랩, 포커스 복귀)

### 메시지 선택 규칙 초안
- `toast`
  - 경미한 알림, 비차단 알림, 짧은 안내
  - 예: 입력 검증 경고, 일시적 네트워크 지연
- `popup`
  - 중요한 실패, 즉시 사용자 인지가 필요한 상황
  - 예: 저장 실패, 권한 오류, 진행 불가 상태
- `auto`
  - 심각도/액션 필요 여부/메시지 길이를 기준으로 `toast` 또는 `popup` 자동 선택

### 완료 판단 기준
- 모든 에러 유형이 `toast/popup/auto` 중 어떤 경로로 노출되는지 규칙이 충돌 없이 정의된 상태

### 목표
- 메시지 시스템을 안정적으로 적용할 수 있도록 Zustand 기반 구조를 설계한다.

### 체크리스트
- [x] 메시지 Store 상태 구조 정의 (`queue`, `defaultMode`, `activePopup`)
- [x] Store 액션 정의 (`pushMessage`, `dismiss`, `clear`, `setMode`)
- [x] 메시지 Payload 스키마 정의
- [x] Renderer 계층 분리 설계 (ToastRenderer / PopupRenderer)
- [x] 도메인 이벤트 -> 메시지 변환 규칙 정의
- [x] 로깅/모니터링 메타데이터 정책 정의 (`source`, `code`)

### Payload 초안
- `id`
- `type`
- `title`
- `description`
- `mode?` (`toast | popup`)
- `duration?`
- `actionLabel?`
- `onAction?`
- `source?`
- `createdAt`

### 완료 판단 기준
